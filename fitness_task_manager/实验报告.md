# 实验报告：基于AI协同开发的任务管理系统设计与实现

## 一、 实验内容、要求或目的

**1. 实验目的：**
* **语言基础与编码能力**：熟悉Go语言在小型控制台系统开发中的应用，掌握核心的语言特性，包括结构体定义（Struct）、切片操作（Slice）、包组织（Packages）等基础程序设计能力。
* **数据持久化机制**：深入理解数据模型与持久化原理，掌握利用Go标准库 `os` 和 `encoding/json` 实现对象数据的序列化内存写入与硬盘文件的映射保存。
* **工程化与AI赋能**：学习并实践利用大语言模型（AI）辅助开发，覆盖从前期的需求分析、系统构建策略到后期的代码排错与重构。
* **逻辑结构延展**：在一套标准CRUD（增删改查）生命周期基础上，结合防篡改记录的密码学“哈希链”思想，完成系统结构与安全模块的综合设计。

**2. 实验任务与内容：**
本次实验采用Go语言设计并实现了一套基于命令行的**“个人健身训练任务管理系统”**。系统重点围绕“任务”实体展开数据的生命周期管理，实现了如下功能：
* **基础功能**：独立设计包含任务ID、名称、描述、状态、创建时间及优先级的完整数据结构。实现健身任务的创建、结构化查询、状态修改以及任务删除。
* **拓展功能**：新增了任务**分类（如：有氧/无氧）**和**截止时间（Deadline）**。
* **加分项实践（防篡改操作追踪）**：独立实现了一套基于“链式日志”的记录系统。每一次影响持久化数据的操作（增/删/改）均会生成一条摘要，附加时间戳并结合上一条记录的哈希针（PrevHash）通过 `SHA-256` 算法加密生成本块数字指纹。最后提供日志独立校验引擎，用于鉴定底层 `json` 文本数据是否曾受到外部非法篡改。

## 二、 AI 参与过程（重点与深度复盘）

由于课堂理论知识吸收存在断层，特别在Go语言特有的数据结构（如切片与指针概念）及持久化操作上不够熟练。本次实验全流程采用“结对编程”模式深度借用了AI辅助，以下是核心的互动与工程化改造复盘：

**1. 需求拆解与架构设计的碰撞：**
* **破冰盲点**：我在初期对Go的面向对象特性感到十分困惑（试图寻找 `Class` 关键字）。我在给AI的 Prompt 中不仅抛出了“做一个带JSON持久化的任务系统”的需求，还特别提出了“我要做一个像区块链一样的操作记录追踪作为加分项，请告诉我如何组织代码”。
* **AI的引导**：AI扮演了架构导师的角色。它不仅向我展示了Go使用 `struct`（结构体）聚合数据的标准范式，更澄清了关于导出的核心机制——为了让 `encoding/json` 标准库能够跨包访问字段进行序列化，**结构体名和内部字段的首字母都必须强制大写**。同时，AI演示了利用反引号配置结构体标签（如 `` `json:"id"` ``）来映射字段名。这段互动不仅敲定了 `FitnessTask` 和 `LogBlock` 的雏形，也彻底解答了我之前的自学疑团。

**2. 核心算法探究、切片内存模型与疑难Bug调试：**
* **踩坑“切片越界与内存泄漏”**：在实现任务删除时，我的初步想法是将目标数组对应的元素赋空值。但这导致了切片存在大量空洞，在后续的遍历与保存中引发了空指针异常崩溃（Panic）。
* **AI的解惑**：AI向我剖析了Go的切片（Slice）底层指向连续内存数组的高级特性。它没有直接包办，而是给出了一段精妙的切片拼接语法 `tasks = append(tasks[:i], tasks[i+1:]...)`。在追问下，我明白了该语法相当于将需要删除的元素前后的两段切片提取闭包，再利用展开操作符 `...` 将后半段底层元素“打散”并重新追加缝合到前半段，实现了优雅且高并发安全的物理删除。
* **攻克哈希加密“雪崩碰撞”Bug**：在实现防篡改记录系统（要求：旧记录被修改，下一次运行必定提示拦截）时，我拼接 `time.Now().String()` 参与哈希运算。却发现即使不篡改文件，每次校验也经常报错“链条断裂”。
* **问题定位**：我将出错代码发给AI，AI帮我指出了致命盲点——不同情况或系统下 `String()` 输出带有微小差异（如毫秒级尾数），导致生成的 SHA-256 指纹引发雪崩效应。在它的建议下，我改用了强制时间戳模板 `time.Now().Format(time.RFC3339)`，成功锁定时间格式，彻底解决了哈希误断裂的Bug。

**3. 代码强力审查与个人重构升级（去“机器味”）：**
* **AI并非工程终点**：初期AI生成的逻辑为了追求直观，在控制台交互里堆砌了大量带有Emoji表情和“讨好型语气”的回显。但这完全不符合一名专业工程师开发 CLI 工具的标准。
* **重构动作一（消除冗余/边界强化）**：我直接“扒掉了它的皮”，亲自重写了全套控制台交互菜单（使其变成严整的制表符排版和干净冷峻的日志文字）。另外，面对 `loadData` 函数中对不存在文件读取直接报错的缺陷，我引入了 `os.IsNotExist` 等规范化错误捕捉，使得系统在初次启动时能平滑生成基础数据表。
* **重构动作二（知识内化/代码归属）**：我主动拷问底盘——比如追问AI `crypto/sha256` 中为何 `Write` 函数必须接收字节序列 `[]byte`。在了解其底层的流式加密后，我删除了AI原有的空泛注释，换上了我自己理解后的中文工程注释。经过这轮极度严格的 Code Review（代码审查），程序代码全方位蜕变成了我个人可控的知识资产。

## 三、 程序主要算法或源代码

### 1. 程序结构说明
本项目代码（`main.go`）结构清晰，无繁冗的深层嵌套，划分为三大模块：
* **数据模型层 (Data Models)**：声明全局实体与对应的JSON映射标签。
* **持久化与安全校验层 (Storage & Ledger)**：处理内存切片和本地文件的 I/O 控制（`loadData`, `saveData`），以及包含密码学算法的审计记录引擎（`recordLog`, `verifyChain`）。
* **业务逻辑层 (Business Logic)**：依托于命令行扫描器 `bufio.NewReader` 的增删改查函数及主循环。

### 2. 核心模块与关键代码分析

**（1）利用 Struct Tag 实现数据模型定义与JSON化：**
Go通过结构体聚合属性，并巧妙使用“标签(Tag)”技术实现了强大的序列化：
```go
// 健身任务核心结构模型
type FitnessTask struct {
	ID          string    `json:"id"`          // 通过Tag标记写入JSON时该字段名
	Name        string    `json:"name"`
	Category    string    `json:"category"`    // 扩展项：健身分类
	Status      string    `json:"status"`
	Priority    int       `json:"priority"`
	CreatedAt   time.Time `json:"created_at"`
}
```

**（2）链式防篡改记录系统（算法核心）：**
利用前置哈希指针，将每次发生的数据变动串联。一旦底层JSON被直接利用记事本强行修改，程序会在校验时拦截。
```go
// 核心哈希算法片段：接收当前时间、摘要内容和前驱节点指纹
func calculateHash(index int, timestamp time.Time, summary, prevHash string) string {
	// 拼接长字符串组合全部信息基料
	record := strconv.Itoa(index) + timestamp.Format(time.RFC3339) + summary + prevHash
	
	h := sha256.New()           // 初始化散列加密对象
	h.Write([]byte(record))     // Go中要求将字符串强转为字节切片 []byte
	return hex.EncodeToString(h.Sum(nil)) // 转产为可视化的16进制指纹
}

// 篡改校验器片段
func verifyChain() bool {
	for i := 1; i < len(blockchain); i++ {
		prev := blockchain[i-1]
		curr := blockchain[i]
		// 规则1：当前链结的“挂钩”是否与上一节实体吻合
		if curr.PrevHash != prev.Hash { return false }
		// 规则2：实体内容是否被恶意更改（用现存数据重新跑一遍散列并对质）
		if curr.Hash != calculateHash(curr.Index, curr.Timestamp, curr.Summary, curr.PrevHash) { return false }
	}
	return true
}
```

## 四、 实验结果与分析

**运行环境**：终端命令行 (`powershell` / `bash`) + `go run main.go`。系统一次性通过内置编译器编译，无任何警告报错。

**系统功能与预期验证：**
1. **初始化与基础功能 (CRUD)**：启动时因根目录无文件，系统顺利初始化“创世区块日志”。调用增删改查测试：控制台展现标准的制表符对齐列表，增删结果准确对应指定ID，数据状态流转完全符合业务预期。
2. **数据持久化验证**：主动中断程序退出进程。查看磁盘工作区，发现已平稳落盘产生 `tasks_data.json` 与 `chain_logs.json` 实体数据文件。再次启动系统，先前写入的健身任务自动恢复展示，持久化验证圆满成功。
3. **加分功能验证（哈希链防篡改检查）**：
   * **未篡改状态**：在系统主菜单按下 `5` 校验引擎，系统回传绿灯提示：“`验证结果：日志链完整，所有历史数据均未被外部篡改！`”。
   * **篡改拦截介入**：将操作内容 `"新增任务: 5公里慢跑"` 偷偷改为 `"删除任务"` 然后保存文件。回到系统的命令行执行检验系统，系统当即进行红灯预警提示：“`警告检测：发现链条断裂或指纹不匹配，记录文件可能已被文本编辑器强行修改过！`”。
   * **分析结论**：哈希校验体系工作十分严密，这印证了微小的数据变动都将引起散列雪崩效应，进而引发连环断链。扩展要求的实现超出预期目标。

**2. 第18题：最大公约数与最小公倍数**
* **核心代码**：
```go
package main

import "fmt"

func gcd(m, n int) int {
	for n != 0 {
		m, n = n, m%n // Go语言多重赋值迭代
	}
	return m
}

func lcm(m, n int) int {
	return (m * n) / gcd(m, n)
}

func main() {
	var m, n int
	fmt.Print("请输入两个正整数 m 和 n: ")
	fmt.Scanf("%d %d", &m, &n)

	fmt.Printf("最大公约数是: %d\n", gcd(m, n))
	fmt.Printf("最小公倍数是: %d\n", lcm(m, n))
}
```

package main

import "fmt"

func evenGenerator() func() int {
	i := 0 // 外部环境变量，由下方闭包捕获生命期

		ret := i
		i += 2 // 每次调用自增保存2

	}
}

func main() {

	fmt.Println(nextEven()) // 4

### 第五次习题练习（第九章：并发编程）

**第9题：协程与通道的随机数通信**
* **代码实现：**
```go
package main

import (
	"fmt"
	"math/rand"
	"time"
)

func main() {
	// 设置随机数种子，确保每次运行结果不同
	rand.Seed(time.Now().UnixNano())

	// 创建一个用于传递整数的通道
	ch := make(chan int)

	// 启动子协程（生产者）
	go func() {
		fmt.Println("[子协程] 开始生成随机数...")
		for i := 0; i < 5; i++ {
			num := rand.Intn(101) // 生成 [0, 100] 的随机数
			ch <- num             // 将数据送入通道
		}
		close(ch) // 关闭通道，通知主协程发送完毕
	}()

	// 主协程（消费者）通过 range 持续接收数据
	fmt.Println("[主协程] 等待接收数据：")
	for val := range ch {
		fmt.Printf("接收到数字: %d\n", val)
	}

	fmt.Println("程序执行完毕。")
}
```
